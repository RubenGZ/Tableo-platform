Arquitectura del sistema

Cómo está construido Tableo

Una explicación visual de las decisiones técnicas que hacen posible la plataforma — sin jerga innecesaria.

1

Las capas del sistema

Tableo funciona como una pila de capas especializadas. Cada capa tiene una función clara y habla solo con la capa de debajo. Así evitamos que todo esté mezclado y el código sea un desastre.

Capa 1 — Lo que ve el usuario
Interfaz Web (Next.js)
Dos experiencias distintas: el panel del negocio (donde el peluquero gestiona su agenda y clientes) y la página pública de reservas (donde el cliente final elige su cita sin necesidad de crear cuenta). Esta capa no contiene ninguna lógica de negocio — solo muestra y recoge información.
↓
Capa 2 — El cerebro
Motor de Disponibilidad
El componente más importante de Tableo. Calcula qué huecos están libres, valida si una reserva es posible y bloquea el slot en la base de datos de forma que dos personas no puedan reservar lo mismo al mismo tiempo. Está diseñado para funcionar con cualquier sector (peluquerías, restaurantes, inmobiliaria) sin cambiar su núcleo.
↓
Capa 3 — Comunicaciones
Notificaciones (Email + SMS)
Se encarga de enviar confirmaciones, recordatorios y alertas tanto al negocio como al cliente. Funciona de forma independiente — si el sistema de notificaciones falla, las reservas siguen funcionando perfectamente.
↓
Capa 4 — La memoria
Base de Datos (Supabase / PostgreSQL)
Donde vive toda la información: negocios, profesionales, reservas y clientes. Tiene una capa de seguridad integrada (RLS) que garantiza que la peluquería A nunca puede ver los datos de la peluquería B, aunque hubiera un bug en el código.
💡

¿Por qué esta estructura? Es lo que se llama un "monolito modular". Todo en un mismo proyecto pero con partes bien separadas. Es la arquitectura de Shopify, GitHub y Basecamp — aguanta millones de usuarios y permite a un equipo de 2 personas moverse muy rápido.

2

El Motor de Disponibilidad

Esta es la pieza más inteligente de Tableo. En lugar de tener un código diferente para cada tipo de negocio, tiene un motor central con "adaptadores" intercambiables — como enchufes de viaje.

⚙️

Motor Central

Sabe hacer tres cosas: ver huecos libres · validar una reserva · bloquear el slot

💇
Adaptador Belleza
Calcula disponibilidad por profesional y duración del servicio. "¿Tiene Laura 60 minutos libres el martes a las 11h?"
✓ V1 — Ya disponible
🍽️
Adaptador Restaurantes
Calcula disponibilidad por mesa y capacidad. "¿Hay mesa para 4 personas el viernes a las 21h?"
V2 — Planificado
🏠
Adaptador Inmobiliaria
Calcula disponibilidad por propiedad. "¿Está libre el piso de Goya para visita el jueves a las 17h?"
V3 — Futuro
🔌

La ventaja clave: Añadir restaurantes en V2 no toca ni una línea del adaptador de belleza. Se crea un archivo nuevo. Esto es lo que evita que el código se convierta en espagueti cuando Tableo crezca a nuevos verticales.

3

Cómo funciona una reserva

El proceso de reserva tiene dos fases para evitar que dos personas reserven el mismo hueco al mismo tiempo. Como cuando compras una entrada de concierto — el asiento se "bloquea" mientras pagas.

1
Ver huecos
El cliente abre el calendario y ve los slots disponibles en tiempo real
2
Bloqueo temporal
Al pulsar un slot, queda bloqueado 5 minutos solo para ese cliente
3
Datos del cliente
Nombre, teléfono, servicio elegido. Sin crear cuenta.
4
Confirmación
La reserva se confirma definitivamente. Email + SMS enviados a ambos.
🔒

¿Qué pasa si dos personas pulsan el mismo slot a la vez? La base de datos tiene un sistema de locks que garantiza que exactamente uno de los dos obtiene la reserva. El otro recibe un mensaje de "Este horario acaba de reservarse" al instante. Cero dobles reservas, garantizado matemáticamente.

⏱

Countdown visible durante el bloqueo: Una vez bloqueado el slot, el cliente ve un contador regresivo en pantalla — "Tienes 4:32 para completar tu reserva". Si el tiempo expira, el slot se libera automáticamente y el calendario se refresca sin recargar la página. Mientras el calendario carga, se muestran bloques animados (skeleton screens) para que la experiencia parezca instantánea.

⏱️

Respuesta en <300ms

Desde que el cliente pulsa "Reservar" hasta que ve la confirmación. Menos de un tercio de segundo.

🔄

Tiempo real

Si alguien reserva un slot, desaparece del calendario de los demás usuarios al instante. Sin recargar la página.

💨

Sin cuenta obligatoria

El cliente final reserva con nombre y teléfono. Sin registro, sin contraseña, sin fricción.

📅

Google Calendar incluido

Al confirmar una reserva, aparece automáticamente en el Google Calendar del negocio. Sin pasos manuales.

¿Qué es el campo metadata? Es un campo flexible que guarda información diferente según el sector. Para una peluquería guarda el servicio y el precio. Para un restaurante guardará el tamaño de la mesa. El schema no cambia — solo cambia lo que va dentro de ese campo.

⏰

timezone_id — todas las fechas en UTC, sin excepción. Este campo indica la zona horaria del negocio (ej. Europe/Madrid, Atlantic/Canary). Todas las fechas se guardan en UTC en la base de datos y solo se convierten a hora local en el momento de mostrarlas. Sin esto, un negocio en Canarias tendría sus citas desplazadas una hora.

🎨

config.branding — la identidad visual del negocio. Dentro del campo config vive la personalización: logo, color de acento ("#e91e8c" para el rosa de una peluquería, "#2e7d32" para el verde de un restaurante ecológico). Los botones de la página de reserva adoptan ese color automáticamente sin reconstruir nada.

5

Seguridad: cada negocio en su burbuja

Todos los negocios comparten la misma base de datos, pero con una separación impenetrable entre ellos. Esta separación vive en la base de datos — ni siquiera un bug en el código puede romperla.

Peluquería "Salon Luna" (sesión activa)
✓ reserva_001 — Ana García — Lunes 10h
✓ reserva_002 — Marta López — Lunes 11h
✓ reserva_003 — Pedro R. — Martes 9h
Ve sus 3 reservas ✓
Peluquería "BarberKing" (otro negocio)
🔒 Acceso bloqueado por la base de datos
reserva_xxx — ██████ ██████
reserva_xxx — ██████ ██████
Datos invisibles para Salon Luna ✗
🛡️

Row Level Security (RLS): Es una funcionalidad de PostgreSQL que filtra automáticamente los datos según quién está conectado. No necesitamos escribir WHERE business_id = X en cada consulta — la base de datos lo hace sola. Si olvidamos el filtro en el código, la base de datos lo aplica igualmente. Seguridad a prueba de descuidos.

Las reglas del código

Para que Claude Code pueda trabajar rápido sin romper cosas, hay reglas claras que todos los archivos deben cumplir.

✅ Sí hacemos esto

✓Un archivo = una responsabilidad clara
✓Archivos de menos de 200 líneas
✓La lógica específica de sector vive en su adaptador
✓Toda tabla nueva tiene RLS activado
✓Two-phase booking para toda reserva
✓Tests por cada adaptador del motor
✓Fechas siempre en UTC — conversión solo en la UI
✓UI de booking específica por sector (nunca genérica)
✓ARIA correcto en todos los componentes de calendario
✓Google Calendar sincronizado al confirmar — no es opcional

❌ Nunca hacemos esto

✗if sector === 'beauty' fuera de los adaptadores
✗Lógica de negocio en los componentes de UI
✗Importar un adaptador directamente desde las páginas
✗Tablas separadas por sector en la DB
✗INSERT directo en bookings sin el sistema de locks
✗Añadir V2 antes de que V1 tenga clientes reales
✗Guardar fechas en hora local en la base de datos
✗Un formulario de reserva igual para todos los sectores
8

UX — Mobile, Branding y Accesibilidad

El 80% de los clientes reservan desde el móvil. El 90% de los dueños gestionan su negocio desde el teléfono. La interfaz tiene que funcionar perfectamente con los dedos.

👆

Calendario táctil

El dashboard detecta si el dispositivo es táctil o de escritorio y carga la variante correcta. En móvil: deslizar para cambiar de semana, mantener pulsado para mover una cita con el dedo.

🎨

Identidad de marca

Cada página de reserva tiene el logo, el color de botones y la imagen del negocio. Una peluquería ve su rosa, un restaurante su verde. Los colores se cargan desde la base de datos, sin reconstruir nada.

♿

Accesible por defecto

Los calendarios son las peores implementaciones de accesibilidad en la web. Tableo usa etiquetas ARIA correctas desde el primer componente. Si falla la accesibilidad, el test de CI falla — no llega a producción.

📅

Google Calendar en el MVP

No es una feature extra: es parte del proceso de confirmación. Al confirmar una reserva en Tableo, aparece automáticamente en el Google Calendar del negocio. Sin esto, los dueños mantienen dos agendas en paralelo y hay overbooking.

📱

La UI es sector-específica aunque el backend sea universal. El formulario de reserva de una peluquería ("elige servicio + profesional") es completamente distinto al de un restaurante ("número de comensales + turno"). Misma API por debajo, experiencia 100% adaptada arriba. Un formulario genérico para todos los sectores haría que la plataforma pareciera mediocre en todos ellos.

9

Stack técnico y autenticación

Las herramientas exactas con las que está construido Tableo. Cada elección tiene una razón concreta — no se usó lo último porque sí.

🔑

Google OAuth — una sola autenticación

El dueño del negocio entra con su cuenta de Google. Ese mismo login da acceso a Tableo y a su Google Calendar. Un clic. No hay contraseña que recordar.

⚡

Server Actions — sin APIs intermedias

Las operaciones (crear reserva, liberar slot) van directamente del formulario al motor, sin pasar por una API intermedia. Menos código, menos puntos de fallo, detección de errores en tiempo de compilación.

🧪

Vitest — tests en milisegundos

El mismo API que Jest pero 10–20× más rápido. Un test suite que tarda 2 segundos en vez de 40 se ejecuta en cada guardado. La accesibilidad se verifica automáticamente — si falla ARIA, el PR no se mergea.

🎨

Tailwind v3 + shadcn/ui

shadcn/ui copia los componentes directamente al repo — sin bundle propio, sin actualizaciones que rompan. Tailwind v4 está aún inestable; v3 es sólido y shadcn lo requiere.

🔒

Por qué Google OAuth y no email+contraseña: Google Calendar es parte del MVP — el token de OAuth sirve para ambas cosas. Con email+contraseña habría que hacer el OAuth de Google de todos modos en un segundo paso. Un solo flujo resuelve las dos necesidades: login del dueño + sincronización de calendario.

📦

pnpm como gestor de paquetes: Hasta 3× más rápido que npm en instalaciones. El lockfile es determinista — dos máquinas distintas instalan exactamente las mismas versiones. En un proyecto con muchas dependencias (Next.js, Supabase, dnd-kit, date-fns…) la diferencia es notable desde el primer día.

En resumen

Las decisiones técnicas que hacen de Tableo una plataforma sólida desde el día 1.

🏗️ Monolito Modular

Un solo proyecto, bien organizado por dentro. Rápido de construir, fácil de mantener para un equipo de 2.

⚙️ Motor Polimórfico

Añadir un nuevo sector es crear un archivo nuevo. Cero impacto en lo que ya funciona.

🗄️ Schema Unificado

5 tablas para todos los sectores. timezone_id + config.branding incorporados desde el inicio.

🔒 Locks Postgres

Garantía matemática de cero dobles reservas. Funciona con cualquier número de usuarios simultáneos.

🛡️ RLS en DB

El aislamiento entre negocios vive en la base de datos. Un bug en el código no puede filtrar datos de otro negocio.

⏰ UTC obligatorio

Todas las fechas en UTC. La zona horaria del negocio solo se usa para mostrar la hora local al usuario.

📱 Mobile-first

Calendario táctil, branding dinámico, countdown de bloqueo, skeleton screens y ARIA desde el primer commit.

🔑 Google OAuth + Server Actions

Un login que da acceso a Tableo y al Google Calendar del negocio. Las operaciones van del formulario al motor sin APIs intermedias.
