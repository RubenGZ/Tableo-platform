🚨 Gaps críticos — bloqueantes antes del scaffold

Decisiones que no se han tomado y que afectan directamente a cómo se escribe el primer archivo de código.

BLOQUEANTE #1 Capa de datos — Server Actions vs API Routes vs tRPC

No hay ADR que decida cómo los componentes de Next.js obtienen y mutan datos. Esta decisión determina la arquitectura de cada componente del proyecto. No se puede escribir ni el primer formulario sin tenerla resuelta.

Opción A — Server Actions: Funciones async en el servidor llamadas directamente desde componentes. Máxima simplicidad, menos boilerplate. Recomendado para equipos pequeños con Claude Code.

Opción B — API Routes (/api/*): Endpoints REST clásicos. Más verboso pero más testeable de forma independiente. Permite reutilizar la API desde la app móvil futura.

Opción C — tRPC: Type-safe end-to-end. Potente pero añade complejidad de setup. Overkill para MVP.

Decisión pendiente: Necesita ADR-008 antes de escribir código.

BLOQUEANTE #2 Autenticación — Flujo exacto de Supabase Auth no definido

Los ADRs mencionan auth.uid() en las políticas RLS pero nadie ha decidido cómo se autentica el dueño del negocio. Supabase Auth soporta múltiples métodos y la elección afecta el wizard de onboarding completo.

Opción A — Email + Password: Clásico. El dueño crea cuenta con email y contraseña. Requiere pantalla de verificación de email.

Opción B — Magic Link (email): El dueño pone su email y recibe un link. Sin contraseña. Más fricción en el primer login, menos gestión de seguridad.

Opción C — Google OAuth: "Continuar con Google". Alta conversión para negocios que ya tienen Google Calendar (que es el caso de Tableo). Combina bien con la integración de Google Calendar.

Decisión pendiente: Necesita ADR-009 antes de construir el onboarding.
