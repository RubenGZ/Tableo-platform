AI Agent API Layer

Referencia técnica completa · Tableo v1.0

Fecha: 2026-04-30 Branch: master 76 tests passing Next.js 15 App Router TypeScript strict

¿Qué es?

La AI Agent API Layer es una capa de integración que permite a cualquier LLM (Gemini, Claude, GPT, Siri, Google Assistant) crear reservas en Tableo usando lenguaje natural. El LLM llama a un único endpoint REST y recibe respuestas JSON diseñadas para ser interpretadas y retransmitidas directamente al usuario final.

Principio de diseño clave: todos los errores incluyen dos campos: message_for_ai (contexto técnico para el LLM) y suggested_user_prompt (frase en español lista para decirle al usuario). El LLM nunca tiene que construir mensajes de error por su cuenta.

Las reservas creadas por IA no se confirman automáticamente. Tienen el status pending_ai_confirmation y el dueño del negocio las aprueba desde el dashboard. Esto garantiza que el negocio siempre tiene el control.

POST /api/ai/book

POST /api/ai/book — Crear reserva desde LLM

Header requerido

x-tableo-ai-token: <TABLEO_AI_TOKEN>

Flujo de 10 pasos

El handler ejecuta siempre los pasos en este orden. Cada paso puede devolver una respuesta de error que cortocircuita el resto.

1
validateAiToken(request)
→ 401 + ERROR_INVALID_TOKEN si el header falta o no coincide con TABLEO_AI_TOKEN
2
request.json()
→ 422 + ERROR_MISSING_DATA si el body no es JSON válido
3
Validar campos obligatorios
Verifica: business_slug, resource_id, datetime, customer.name
→ 422 + ERROR_MISSING_DATA con array missing[] de campos ausentes
4
Fetch business por slug
SELECT id, name, sector_type, timezone_id, config FROM businesses WHERE slug = ?
→ 404 + ERROR_BUSINESS_NOT_FOUND si no existe
5
normalizeDateTime(body.datetime, business.timezone_id)
Convierte cualquier formato al Date UTC de startAt
→ 422 + ERROR_INVALID_DATETIME si no puede parsear
6
getAdapter(sector).getSlots(resourceId, dateStr, timezoneId)
Obtiene slots disponibles y busca si el solicitado existe (tolerancia 1 minuto)
→ 409 + ERROR_CAPACITY_FULL con hasta 3 alternative_slots si el slot no está disponible
7
Upsert customer
Busca cliente por phone OR email dentro del negocio. Si no existe, lo crea.
→ 422 si el insert falla
8
Verificar reserva duplicada
SELECT id FROM bookings WHERE customer_id = ? AND resource_id = ? AND overlap(startAt, endAt) AND status IN (reserved, confirmed, pending_ai_confirmation)
→ 409 + ERROR_DUPLICATE_BOOKING si existe solapamiento
9
INSERT booking
status: 'pending_ai_confirmation', ai_source: true, metadata: {service, notes, ai_requested_by}
→ 500 + ERROR_SERVER si el insert falla
✓
Return 201 + AiSuccessResponse
booking_id, status, start_at, end_at, message_for_ai, suggested_user_prompt

Módulo: validateAiToken()

Archivo: src/lib/ai/token-auth.ts

ParámetroTipoDescripción
requestRequestRequest HTTP de Next.js
Retornaboolean — true si el token es válido

Comportamiento:

  • Lee el header x-tableo-ai-token del request
  • Compara con process.env.TABLEO_AI_TOKEN usando crypto.timingSafeEqual (previene timing attacks)
  • Si el env var no está configurado → loguea error y devuelve false
  • Si el token falta en el header → devuelve false inmediatamente
import { timingSafeEqual } from 'crypto'

export function validateAiToken(request: Request): boolean {
  const token    = request.headers.get('x-tableo-ai-token')
  const expected = process.env.TABLEO_AI_TOKEN

  if (!expected) { console.error('[AI API] TABLEO_AI_TOKEN env var not set'); return false }
  if (!token) return false

  const a = Buffer.from(token)
  const b = Buffer.from(expected)
  return a.byteLength === b.byteLength && timingSafeEqual(a, b)
}

Módulo: normalizeDateTime()

Archivo: src/lib/ai/date-normalizer.ts · Dependencias: chrono-node, @date-fns/tz

ParámetroTipoDescripción
inputstring | {date, time}Input del LLM — cualquier formato
timezoneIdstringIANA timezone del negocio (ej. "Europe/Madrid")
referenceDateDate?Fecha base para "mañana", "el viernes", etc. Default: new Date()
RetornaDate en UTC · Lanza ParseError si no puede parsear o la fecha relativa está en el pasado

4 ramas de parseo (en orden)

Rama 1 — Objeto {date, time}

Formato de Siri y Google Assistant. Combina las partes en un ISO local y lo convierte a UTC usando el TZDate de @date-fns/tz.

{ date: "2024-06-17", time: "10:00" }  // → new TZDate(2024, 5, 17, 10, 0, 0, 'Europe/Madrid') → 08:00 UTC
{ date: "2024-06-17", time: "15" }     // time sin minutos → se normaliza a "15:00"
Rama 2 — ISO 8601 con timezone explícita

Termina en Z, +HH:MM o -HH:MM. Se pasa directamente a new Date() sin transformación.

"2024-06-17T10:00:00Z"         // UTC directo
"2024-06-17T10:00:00+02:00"    // Madrid verano → 08:00 UTC
Rama 3 — ISO sin timezone

Se interpreta como hora local del negocio. Usa el constructor por partes de TZDate para evitar que el servidor lo interprete en su propia zona horaria.

"2024-06-17T10:00:00"  // → TZDate(2024, 5, 17, 10, 0, 0, business.timezone_id)
Rama 4 — Texto relativo (chrono-node)

Usa chrono.parseDate (inglés) primero, luego chrono.es.parseDate (español). Pasa el offset UTC de la timezone del negocio para que mañana a las 10 se interprete en la hora del negocio, no del servidor.

"mañana a las 10"           // chrono.es — español
"tomorrow at 3pm"           // chrono   — inglés
"el viernes a las 3 de la tarde"   // próximo viernes 15:00 local
Solo la rama 4 rechaza fechas pasadas. Los ISO explícitos se aceptan siempre — la validación de pasado es responsabilidad del caller.

Códigos de error

ERROR_CAPACITY_FULL

HTTP 409 · El slot solicitado no está disponible

"Lo siento, a las 10:00 ya no hay disponibilidad en Salón Luna. ¿Te vendría bien a las 10:30, 11:00 o 11:30?"
ERROR_MISSING_DATA

HTTP 422 · Faltan campos obligatorios

"Para hacer la reserva necesito que me indiques: business_slug, customer.name."
ERROR_INVALID_DATETIME

HTTP 422 · No se pudo parsear la fecha

"No he entendido bien la fecha 'el día ese'. ¿Puedes decirme el día y la hora con más detalle?"
ERROR_DUPLICATE_BOOKING

HTTP 409 · El cliente ya tiene cita en ese horario

"Parece que María ya tiene una cita en Salón Luna a esa hora. ¿Quieres cambiarla o es para otra persona?"
ERROR_BUSINESS_NOT_FOUND

HTTP 404 · El slug no corresponde a ningún negocio

"No he encontrado el negocio 'salon-xyz'. ¿Puedes confirmar el nombre exacto?"
ERROR_INVALID_TOKEN

HTTP 401 · Token ausente o incorrecto

"No tengo autorización para hacer reservas. Por favor, contacta directamente con el negocio."

Diseño conversacional de errores

Cada error tiene dos campos de texto con audiencias distintas:

message_for_ai

Técnico, para el LLM. Explica qué pasó para que el LLM pueda tomar decisiones.

"The requested slot 10:00 is not
available for the requested resource.
3 alternative slots found."

suggested_user_prompt

En español, para el usuario final. El LLM puede decirlo directamente.

"Lo siento, a las 10:00 ya no hay
disponibilidad en Salón Luna.
¿Te vendría bien a las 10:30?"

Migration 007

Archivo: supabase/migrations/007_ai_booking.sql

-- 1. Ampliar CHECK de status
ALTER TABLE bookings DROP CONSTRAINT IF EXISTS bookings_status_check;
ALTER TABLE bookings ADD CONSTRAINT bookings_status_check
  CHECK (status IN (
    'pending', 'reserved', 'confirmed', 'cancelled',
    'completed', 'no_show', 'disputed', 'pending_ai_confirmation'
  ));

-- 2. Columna de origen IA
ALTER TABLE bookings
  ADD COLUMN IF NOT EXISTS ai_source boolean NOT NULL DEFAULT false;

-- 3. Índice parcial para el dashboard
CREATE INDEX IF NOT EXISTS idx_bookings_ai_pending
  ON bookings (ai_source, status)
  WHERE ai_source = true AND status = 'pending_ai_confirmation';
