Goals — El Reto de la Reserva Universal

¿Cómo unificamos lógicas radicalmente distintas en un solo modelo?

⚠️
El fallo más común al diseñar plataformas "universales": intentar construir todo a la vez y acabar con una plataforma mediocre en todos los verticales. Tableo debe ser excelente en Belleza & Bienestar antes de expandirse. La arquitectura debe ser extensible, no la implementación.

Los 3 tipos de reserva y su primitiva común

⏱ Basada en Tiempo

Ejemplos: Peluquerías, spas, médicos

  • Recurso = profesional disponible
  • Restricción = duración del servicio
  • Variable = múltiples profesionales simultáneos

V1 — Implementar ahora

🪑 Basada en Inventario

Ejemplos: Restaurantes, hoteles

  • Recurso = mesa o habitación
  • Restricción = capacidad + turnos
  • Variable = tamaño del grupo

V2 — Diseñar ahora, implementar luego

🏠 Basada en Activos

Ejemplos: Inmobiliaria, coches

  • Recurso = activo único
  • Restricción = ocupación exclusiva
  • Variable = ventanas de inspección

V3 — Solo arquitectura extensible

💡
La primitiva unificadora: Los tres tipos comparten el mismo átomo de reserva: Slot(resource_id, start_time, end_time, capacity, status). La diferencia está solo en cómo se calcula la disponibilidad y qué significa "recurso" en cada sector. Esta es la base del Motor Polimórfico.

Modelo de datos unificado (sector-agnostic)

-- Core: sector-agnostic. Nunca cambia entre verticales. CREATE TABLE businesses ( id uuid PRIMARY KEY, name text, sector_type text -- 'beauty' | 'restaurant' | 'real_estate' config jsonb -- configuración específica del sector ); CREATE TABLE resources ( id uuid PRIMARY KEY, business_id uuid REFERENCES businesses, resource_type text, -- 'staff' | 'table' | 'asset' name text, metadata jsonb -- atributos específicos: {'specialties': ['corte', 'color']} ); CREATE TABLE bookings ( id uuid PRIMARY KEY, resource_id uuid REFERENCES resources, customer_id uuid, start_at timestamptz, end_at timestamptz, status text, -- 'pending' | 'confirmed' | 'cancelled' | 'completed' metadata jsonb -- {'service': 'corte', 'party_size': 4, ...} ); CREATE TABLE availability_windows ( id uuid PRIMARY KEY, resource_id uuid REFERENCES resources, day_of_week int, -- 0=Lun, 6=Dom open_time time, close_time time );
✅
Por qué este diseño es correcto: El campo metadata jsonb en resources y bookings absorbe todos los atributos sector-específicos sin romper el esquema. Añadir el vertical de restaurantes en V2 no requiere migración — solo un nuevo adaptador.

El problema de concurrencia — y cómo resolverlo

⚡
Escenario de riesgo: El último slot disponible un sábado a las 11:00h en una peluquería popular. 3 clientes abren la página simultáneamente y pulsan "Reservar" al mismo tiempo. Sin control de concurrencia → 3 reservas en el mismo slot → caos.
-- Solución: Two-Phase Booking con SELECT FOR UPDATE NOWAIT -- Esto es un primitivo de PostgreSQL — funciona perfecto en Supabase -- FASE 1: Reserva temporal (TTL 5 minutos) BEGIN; SELECT * FROM slots WHERE resource_id = $1 AND start_at = $2 AND status = 'available' FOR UPDATE NOWAIT; -- Rechaza instantáneo si ya está bloqueado UPDATE slots SET status = 'reserved', reserved_until = NOW() + INTERVAL '5 minutes', reserved_by = $session_id WHERE id = $slot_id; COMMIT; -- FASE 2: Confirmar (tras pago o confirmación del usuario) UPDATE bookings SET status = 'confirmed' WHERE session_id = $session_id AND reserved_until > NOW(); -- pg_cron limpia reservas expiradas cada minuto SELECT cron.schedule('cleanup-expired', '* * * * *', 'UPDATE slots SET status=''available'' WHERE status=''reserved'' AND reserved_until < NOW()');

Architecture — Monolito Modular + Motor Polimórfico

Cómo evitar el código espagueti sin sobre-ingeniería prematura

❌
El anti-patrón que destruye plataformas universales: if (sector === 'beauty') {'{'} ... {'}'} else if (sector === 'restaurant') {'{'} ... {'}'} en el núcleo del sistema. Cada nuevo sector añade 200 líneas de condicionales. En 6 meses nadie entiende el código. Claude Code empieza a romper cosas al editar.

El Motor de Disponibilidad Polimórfico

Estructura de directorios del proyecto

src/ ├── app/ next.js app router │ ├── (dashboard)/ panel del negocio (auth required) │ ├── (booking)/[slug]/ página pública de reserva (no auth) │ └── api/ │ └── availability/ API route — llama al engine │ ├── availability/ EL MOTOR — núcleo del sistema │ ├── engine.ts interfaz + factory — NUNCA importa adapters directamente │ ├── types.ts Slot, BookingRequest, AvailabilityAdapter │ └── adapters/ │ ├── beauty.adapter.ts lógica específica de belleza — V1 │ └── restaurant.adapter.ts V2 — no existe aún, no contamina V1 │ ├── modules/ features por dominio │ ├── businesses/ │ ├── bookings/ │ ├── customers/ │ ├── notifications/ email + SMS — sector-agnostic │ └── payments/ stripe — sector-agnostic │ └── lib/ utilidades compartidas ├── supabase/ └── utils/
✅
Por qué esta estructura es AI-friendly: Claude Code puede editar beauty.adapter.ts sin tocar el engine. Puede añadir restaurant.adapter.ts sin romper nada existente. Cada archivo tiene una responsabilidad clara que cabe en el contexto de un prompt. Sin this, Claude empieza a cometer errores al editar archivos de 500+ líneas con lógica mezclada.

Reglas de oro del Monolito Modular

✅ Lo que SÍ hacer

  • Un módulo, una responsabilidad
  • Los adaptadores implementan una interfaz común
  • El engine selecciona el adaptador por business.sector_type en un factory
  • Los módulos se comunican solo a través de sus interfaces públicas
  • Tests unitarios por adaptador — aislados

❌ Lo que NO hacer

  • Condicionales if sector === 'x' fuera de los adaptadores
  • Importar un adaptador directamente desde pages/api
  • Mezclar lógica de negocio en los API routes
  • Tablas separadas por sector en la base de datos
  • Añadir V2 antes de que V1 esté en producción con clientes reales

Veredicto Crítico — Los 3 Riesgos Técnicos Más Altos

Los fallos que hunden plataformas de reservas. Y cómo prevenirlos.

1
Double-booking bajo carga concurrente
Disponibilidad & Concurrencia
Severidad: CRÍTICA
El escenario

Una peluquería popular tiene el último slot del sábado. 3-5 usuarios llegan simultáneamente a la página de reserva. Todos ven el slot disponible (la disponibilidad se consultó hace 2 segundos). Los 3 pulsan "Confirmar" en el mismo segundo. Sin control de concurrencia, los 3 obtienen confirmación. El lunes el negocio recibe 3 clientes para el mismo horario.

Impacto real

Un solo doble-booking publicado en redes sociales puede destruir la confianza en Tableo. Los negocios de peluquería viven de su reputación local. Este tipo de incidente se viraliza en grupos de WhatsApp del barrio. No hay segunda oportunidad.

Solución de ingeniería

Two-Phase Booking con SELECT FOR UPDATE NOWAIT de PostgreSQL. Fase 1: reserva temporal con TTL de 5 minutos usando un lock a nivel de fila (no a nivel de aplicación). El primer usuario que ejecuta la transacción gana el lock; los demás reciben un error inmediato y se les muestra "Este slot acaba de ser reservado". Fase 2: confirmación definitiva tras acción del usuario. pg_cron limpia reservas temporales expiradas cada minuto. Este patrón es robusto porque el lock vive en la base de datos, no en memoria de la aplicación — funciona incluso con múltiples instancias de Next.js.

2
Spaghetti arquitectónico al escalar sectores
Arquitectura & Mantenibilidad
Severidad: ALTA
El escenario

En el mes 4, hay demanda de restaurantes. Se añade lógica de mesas directamente en el availability engine con if (sector === 'restaurant'). En el mes 7 llega inmobiliaria. Para el mes 10, el archivo engine.ts tiene 800 líneas con condicionales anidados. Claude Code empieza a romper V1 (beauty) al intentar añadir features de V3 (real estate). El equipo de 2 ya no puede mantener el ritmo.

Impacto real

Velocidad de desarrollo cae un 70%. El tiempo de Claude Code en generar código correcto se multiplica porque los archivos son demasiado largos y complejos para razonar en un solo contexto. La deuda técnica se convierte en el principal bloqueante del negocio.

Solución de ingeniería

Motor de Disponibilidad Polimórfico con registro de adaptadores. El engine nunca importa los adaptadores directamente. Los adaptadores se registran a sí mismos en un factory al inicializar la aplicación. El engine solo conoce la interfaz AvailabilityAdapter. Añadir un sector nuevo = crear un archivo nuevo que implementa la interfaz. Cero cambios en el engine existente. Los archivos se mantienen cortos (<200 líneas), Claude Code puede razonar sobre cada uno de forma aislada, y los tests unitarios por adaptador garantizan que V1 no rompe al añadir V2.

3
Fricción de migración mata la adopción
Product & Go-to-Market
Severidad: ALTA
El escenario

Una peluquería lleva 3 años en Booksy. Tiene 400 clientes con historial de visitas, preferencias, notas internas. El proceso de cambio a Tableo requiere: exportar CSV de Booksy, reformatear manualmente, subir a Tableo, avisar a 400 clientes de que cambien de plataforma. Tiempo estimado: 4-6 horas. Resultado: el 90% de los negocios interesados decide no migrar.

Impacto real

El mejor producto del mundo no crece si la fricción de entrada es demasiado alta. Booksy tiene el lock-in de los datos como su mayor ventaja competitiva, no sus features. Si Tableo no resuelve esto, el TAM accesible se reduce a negocios nuevos que nunca han usado nada (más lentos de convertir y con menos datos).

Solución de ingeniería

Migration Wizard + White-Glove para los primeros 50. Técnicamente: importador CSV de Booksy (el formato es conocido y documentado), sincronización bidireccional con Google Calendar (para negocios que usan GCal como sistema primario), y detección automática de duplicados en la importación. No técnicamente: para los primeros 50 negocios, los founders hacen la migración de forma manual — esto no escala pero genera un aprendizaje invaluable sobre los puntos de dolor reales, y esos 50 negocios se convierten en los mejores embajadores del producto. Después de 50, el proceso es lo suficientemente conocido para automatizarlo completamente.

Auditoría — 3 Puntos Críticos a Corregir

Revisión externa del plan original. Correcciones incorporadas al diseño.

A
El "Veneno" de la Generalización Temprana en la UI
Backend unificado ≠ UI genérica
Corrección incorporada
El problema

El schema de base de datos es correctamente unificado, pero si el formulario de reserva es el mismo para una peluquería y para una inmobiliaria, el resultado es una interfaz confusa y sin personalidad. Un cliente que reserva una visita a un piso necesita ver campos completamente distintos al que reserva un corte de pelo.

Corrección — UI Components específicos por sector

Aunque todos los adaptadores del Motor de Disponibilidad comparten la misma interfaz de backend, cada sector tiene sus propios componentes de UI específicos. El formulario de reserva de belleza muestra "Elige servicio + profesional". El de restaurante muestra "Número de comensales + ocasión". El de inmobiliaria muestra "Tipo de visita + nombre del agente". Misma API, experiencia completamente diferente.

// src/app/(booking)/[slug]/page.tsx // El componente de reserva se selecciona por sectorType — nunca es genérico import { BeautyBookingForm } from '@/modules/booking/ui/beauty' import { RestaurantBookingForm } from '@/modules/booking/ui/restaurant' const BOOKING_FORMS = { beauty: BeautyBookingForm, restaurant: RestaurantBookingForm, real_estate: RealEstateBookingForm, } export default function BookingPage({ business }) { const Form = BOOKING_FORMS[business.sector_type] return <Form business={business} /> // UI 100% específica por sector }
B
El Problema de las Zonas Horarias
Fallo catastrófico no mencionado en el análisis original
Riesgo crítico añadido
El problema

El análisis original no mencionó las zonas horarias. Si un negocio está en Canarias (UTC+0 en invierno) y el servidor procesa la reserva asumiendo Madrid (UTC+1), todas las citas se desplazan una hora. Para un sistema de reservas esto es un fallo catastrófico: el cliente llega una hora antes o una hora después. Especialmente crítico al escalar a Portugal, Latinoamérica o cualquier zona fuera de Madrid.

Corrección — UTC en DB + timezone_id por negocio

Regla absoluta: todas las fechas se almacenan en UTC en la base de datos. Nunca en hora local. El campo timezone_id en la tabla businesses (ej. 'Europe/Madrid', 'Atlantic/Canary') se usa para convertir UTC a hora local solo en el momento de mostrar al usuario. El Motor de Disponibilidad opera siempre en UTC internamente.

-- businesses ahora incluye timezone_id (ADR-003 actualizado) ALTER TABLE businesses ADD COLUMN timezone_id text NOT NULL DEFAULT 'Europe/Madrid'; -- Ejemplos: 'Atlantic/Canary' | 'America/Mexico_City' | 'Europe/Lisbon' -- Todas las fechas en bookings son timestamptz (UTC automático en Postgres) -- La conversión a hora local ocurre SOLO en la capa de presentación: // src/lib/dates.ts export function toLocalTime(utcDate: Date, timezoneId: string): string { return utcDate.toLocaleString('es-ES', { timeZone: timezoneId }) }
C
Google Calendar — De Fase 2 a Skill Central del Motor
Prioridad mal asignada en el plan original
Prioridad reclasificada
El problema

El análisis original relegó la sincronización con Google Calendar a "Fase 2". Esto es un error estratégico: los dueños de negocios pequeños no mirarán el dashboard de Tableo, mirarán su teléfono. Si una cita confirmada en Tableo no aparece en su Google Calendar, el negocio gestionará su agenda manualmente en paralelo — y habrá overbooking. Para hostelería e inmobiliaria esto no es una mejora, es un requisito de día 1.

Corrección — Google Calendar como Skill del Motor de Disponibilidad

La sincronización bidireccional con Google Calendar (y Outlook) pasa a ser una capability integrada en el Motor de Disponibilidad, no un módulo externo opcional. Cuando el Motor confirma una reserva, dispara automáticamente la sincronización con el calendario externo del negocio. Cuando se detecta un evento en Google Calendar que colisiona con un slot disponible, el Motor lo bloquea automáticamente. Esto se implementa en el MVP de V1 para el vertical de Belleza.

// src/availability/adapters/beauty.adapter.ts // Tras confirmar una reserva, el adaptador dispara la sync async confirmBooking(bookingId: string): Promise<void> { await db.bookings.update({ status: 'confirmed' }, bookingId) // Sync bidireccional — no es opcional, es parte del confirm await calendarSync.pushEvent({ bookingId, calendarId: business.googleCalendarId, title: `Reserva: ${service.name} — ${customer.name}`, start: booking.start_at, end: booking.end_at, timezone: business.timezone_id // punto B aplicado aquí también }) }
📋
Impacto en el roadmap: Google Calendar API (OAuth 2.0) se añade al MVP de V1. Outlook Calendar en V2. La integración requiere que el negocio conecte su cuenta de Google durante el onboarding — esto se convierte en un paso del wizard de alta, no en una feature opcional.

Veredicto Final

Tableo es técnicamente viable con el stack y enfoque propuesto. El mayor riesgo no es la tecnología — es la ejecución disciplinada: arrancar en un solo vertical, proteger la arquitectura desde el día 1, y resolver la migración antes de escalar. Con 2 personas + Claude Code y un Monolito Modular bien diseñado, el MVP puede estar en producción en 4-6 semanas con margen operativo positivo desde el día 5.
